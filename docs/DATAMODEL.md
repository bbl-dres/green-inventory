# Green Inventory — Data Model

This document describes the data flow from the source FileGDB
(`GFMBundesgaerten.gdb`) through the conversion script
([`scripts/gdb_to_geojson.py`](../scripts/gdb_to_geojson.py)) into the
single output file the web app consumes
([`data/data.geojson`](../data/data.geojson)). It also documents the
codelists, coordinate systems, geometry-processing rules, and the
frontend data contract.

It is descriptive — generated by reading the current code and source GDB,
not the other way around. When the code changes, update this file.

### 2.5 Critical insight: `fk_profil` is two distinct domains

`fk_profil` on `Pflegeelement_point` references **`idPP`**
(23 entries: trees + small structures + furniture).
`fk_profil` on `Pflegeelement_polygon` references **`idPPy`**
(44 entries: vegetation + surfaces + water).

Profile **1** is `Laubb. nat. grossk.` for points but `Geb.Rasen kf.`
for polygons. The two domains share integer codes but mean different
things — never merge them. The conversion script and frontend keep
them separate via `entity_type`.

## 4. Conversion pipeline

[`scripts/gdb_to_geojson.py`](../scripts/gdb_to_geojson.py)

### 4.1 Steps

1. **Verify source CRS.** `pyogrio.read_info(GDB, layer="Objekt")` is
   asserted to return one of `EPSG:21781` or `EPSG:2056`. Refuses to
   run otherwise — this guards against silent double-reprojection
   if a future GDB drop changes CRS.
2. **Build CRS transformers.** `_best_transformer()` enumerates
   `pyproj.transformer.TransformerGroup` and selects the lowest
   `accuracy` operation. Logs the operation chain and accuracy at
   start.
3. **Extract codelists** via `extract_codelists()` — see §3.
4. **Load `Objekt`** — site polygons (73). Each becomes two output
   features (a polygon + a centroid point). Site fields are
   denormalised onto every child feature.
5. **Load `Pflegeelement_point`** (2 737) — split into trees vs other
   points based on whether `baumart` text or `fk_baumart` integer
   is populated.
6. **Load `Pflegeelement_polygon`** (3 283) — split into tree-canopy
   circles vs regular area polygons based on a circularity heuristic
   (see §6.3).
7. **Assemble output** — order features site → area → canopy → tree
   → point → site_location, assign `id = array_index`.
8. **Validate + clean** — recursive walk replaces any stray NaN
   with `null`, then `json.dump(allow_nan=False)`.

### 4.2 Dependencies

```
pyogrio    0.12+   (FileGDB driver, full-schema reader)
pyproj     3.x     (CRS transforms; uses bundled CHENyx06 grid)
shapely    2.x     (geometry ops, validity, simplify, orient)
pandas     1.x     (Excel codelist read — legacy fallback only)
```

GDAL itself is bundled with pyogrio (pyogrio.libs); we load it
directly via `ctypes` for the field-domain API which pyogrio doesn't
expose.

## 5. Output schema

Single file: [`data/data.geojson`](../data/data.geojson) (~10 MB).

```jsonc
{
  "type": "FeatureCollection",
  "bbox": [west, south, east, north],         // RFC 7946 §5
  "metadata": { ... },                         // see §5.1
  "features": [ ... ]                          // see §5.2
}
```

**RFC 7946 compliance:** no `crs` member; coordinates are WGS84
(longitude, latitude); right-hand winding rule applied to all
polygons.

### 5.1 `metadata` block

| Key | Type | Description |
|---|---|---|
| `source` | string | Absolute path of the source GDB at conversion time |
| `attribution` | string | "© Bundesamt für Bauten und Logistik (BBL) — Bundesgärtnerei" |
| `license` | string | Currently "Internal use only" |
| `extracted_at` | ISO-8601 | UTC timestamp of conversion run |
| `src_crs` | string | "EPSG:21781" or "EPSG:2056" |
| `out_crs` | string | "EPSG:4326 (WGS84, RFC 7946)" |
| `transform_accuracy_m` | object | Published accuracy in metres for each leg of the reprojection chain |
| `simplify_tolerance_m_lv95` | number | DP tolerance applied to outlier polygons (5 cm) |
| `simplify_vertex_threshold` | int | Minimum vertex count to trigger simplification (200) |
| `canopy_min_area_m2` | number | Threshold below which "circular polygons" are treated as decorative, not tree crowns (1.5 m²) |
| `canopy_circularity_tol` | number | Circularity ratio tolerance for canopy detection (0.08) |
| `lv95_in_properties` | bool | true — every feature carries Swiss-grid coordinates as properties |
| `codelists` | object | Verbatim copy of all 17 GDB field domains. Format `{ domainName: { code: label } }` |
| `counts` | object | `{ sites, site_locations, trees, points_other, tree_canopies, areas, total }` |
| `fk_profil_values` | int[] | Distinct `fk_profil` integers seen across all features |

> The heuristic identifies them by circularity, not semantics.
> See §6.3.

Point. Tree positions from `Pflegeelement_point` (those with


#### `entity_type: "point"` (1 116 features)

Point. Non-tree points from `Pflegeelement_point` — furniture,
benches, lamps, ornament, small structures. Same property set as
`tree` minus the species fields.

## 6. Coordinate systems & accuracy

### 6.1 Reprojection chain

The pipeline goes **LV03 → LV95 → WGS84**. Two-step instead of direct
because the LV03↔LV95 step uses the CHENyx06 grid (sub-decimetre) and
gives us a metric LV95 representation we can use for area calculations
and metric-tolerance simplification.

| Leg | Method | Published accuracy |
|---|---|---:|
| `EPSG:21781` → `EPSG:2056` (LV03 → LV95) | CHENyx06 NTv2 grid (bundled with pyproj-data) | **0.2 m** |
| `EPSG:2056` → `EPSG:4326` (LV95 → WGS84) | 7-parameter Helmert | **1.0 m** |
| Direct LV03 → WGS84 | Composed (1.5 m) | (not used) |

Sub-decimetre WGS84 would require swisstopo's Reframe Web API
(grid-based per request). For an inventory app, ~1 m is acceptable.

### 6.2 Coordinate precision in output

All WGS84 coordinates are rounded to **7 decimal places** =
~1.1 cm at 47°N latitude / ~0.8 cm in longitude. Source data is
cm-precision, so this is a faithful round-trip.

Each feature also carries Swiss-grid coordinates as properties:

| Property | Entity types | Precision |
|---|---|---|
| `lv95_east`, `lv95_north` | `tree`, `point` | 0.01 m |
| `lv95_east_centroid`, `lv95_north_centroid` | `site`, `site_location` | 0.01 m |

LV95 is the canonical reference for Swiss surveyors / BBL staff.
The frontend footer shows both WGS84 and LV95 live as the cursor moves.

### 6.3 Identify-on-click in the frontend

When a feature is identified via swisstopo's MapServer, the click
point and `mapExtent` are converted to LV95 client-side using the
swisstopo "Reframe" closed-form polynomial approximation (see
[`js/map.js`](../js/map.js) — `wgs84ToLv95`/`lv95ToWgs84`,
sub-decimetre accurate inside Switzerland). This is needed because
WMS-backed identify endpoints (e.g. `ch.swisstopo-vd.stand-oerebkataster`)
return zero hits when queried with `sr=4326` but work in `sr=2056`.

## 7. Geometry processing

### 7.1 Validity and orientation

Every polygon goes through `clean_polygon()` ([gdb_to_geojson.py:128](../scripts/gdb_to_geojson.py)):

1. `make_valid()` if `geom.is_valid` is false. If the result is a
   `GeometryCollection`, the largest polygon part is kept.
2. **Conditional simplification** — Douglas-Peucker at 5 cm tolerance
   in LV95 metres, applied **only when vertex count > 200**. This
   trims the digitization-noise outliers (originally up to 41 823
   vertices on a 1 060 m² polygon) while keeping normal polygons
   pixel-perfect at any zoom. Tree-canopy candidates are skipped from
   simplification (they're regular polygons by design and we don't
   want to flatten the curvature).
3. `orient(sign=1.0)` — RFC 7946 §3.1.6 right-hand rule (outer rings
   CCW, inner rings CW). Required for Turf.js / D3-geo / leaflet-pip
   point-in-polygon tests; MapLibre is forgiving but other tools aren't.

### 7.2 Vertex statistics

| | original GDB | after pipeline |
|---|---:|---:|
| Total vertices | 304 511 | **170 102** |
| Max polygon | 41 823 | **1 321** |
| Median polygon | 16 | 16 (unchanged) |
| File size | 25.7 MB (raw) | **16.1 MB** |

### 7.3 Tree-canopy circle detection

Polygons in `Pflegeelement_polygon` whose perimeter and area satisfy
the regular-polygon-as-circle relation `P²/(4πA) ≈ 1` are classified
as canopies. Two thresholds gate this:

```python
CANOPY_MIN_AREA_M2 = 1.5
CANOPY_CIRCULARITY_TOL = 0.08
```

The min-area check rejects ornamental Bollensteine (round paving
stones, ~0.05 m² each) which are also 12-gon circles in the source
data. The 8 % circularity tolerance accepts 12-gons (typical),
decagons, and octagons; it rejects squares (ratio 1.273).

## 8. Frontend integration

### 8.1 Data load

`map.js` fetches `data/data.geojson` once on map load. The result is
held in `geojsonData` in module scope. Every feature gets
`f.id = arrayIndex` so URL `?sel=N` deep-links work.

### 8.2 MapLibre source config

```js
map.addSource('features', {
  type: 'geojson',
  data: geojsonData,
  tolerance: 0,        // no tile-level Douglas-Peucker
  maxzoom: 22,         // re-tile up to map's max zoom
});
```

`tolerance: 0` because pipeline-side simplification is already
applied conservatively (see §7.1) — letting MapLibre re-simplify
adds visible angularity at z21+ on the aerial basemap.

### 8.5 Legend grouping

[`js/config.js`](../js/config.js) `LEGEND_GROUPS` maps each top-level

original PDF Grünflächenpflege legend (Rasen / Wiesen / Rabatten /
Hecken / Gehölzflächen / Spez. Bepflanzungsformen / Beläge /
Wasserflächen / Anderes) and is the single source of truth for both
legend rendering and layer-filtering (eye toggles).

### 8.6 Filter sidebar

Right-side `<aside id="filter-sidebar">` panel. Filter columns and
their order live in `FILTER_COLS_DEFAULT`:


## 9. Known limitations

- **WGS84 accuracy bottoms out at ~1 m** because no improved grid
  exists in PROJ's database for `CH1903+ → WGS 84`. Sub-decimetre
  needs the swisstopo Reframe Web API (out of scope).
- **`fk_zustand` and `fk_kostenstelle` codelists are empty** in the
  GDB. The fields exist on every row but values are always 0.
- **`idPL` codelist is empty** — likely reserved for a linear-feature
  layer that doesn't exist in this GDB drop.
- **`tree_canopy` entity-type label is misleading.** Most circular
  polygons under the heuristic are decorative shrub plantings (Profil
  19 "Gehölzrabatte"), not actual tree crowns. Renaming to
  `circular_shrub` or similar is a backlog item.
- **Map polygons render solid only**, even where the legend shows
  patterns (Magerrasen with brown dots, Wechselflor with magenta dots,
  Rasengittersteine with cross-hatch). MapLibre's `fill-pattern` would
  resolve this but requires sprite raster generation.
- **`Strap./Sportr.`, `Sand`, `Kunstrasen`, `Steildach`, `Dachterrasse`,
  furniture point profiles, and `Nestkasten`** don't appear in the
  source PDF legend; their rendering colours are extrapolated to
  nearby PDF colours.
- **No 3D mesh for buildings on the swisstopo aerial basemap** — the
  building extrusion currently uses OpenFreeMap (OSM-derived). For a
  pure swisstopo experience, swissBUILDINGS3D vector tiles would need
  to be wired in.

## 10. Re-running the pipeline

```sh
python scripts/gdb_to_geojson.py
```

Re-reads the GDB at `Daten_Bundesgärtnerei/GFMBundesgaerten.gdb`
(path hardcoded — change `GDB_PATH` constant if different),
overwrites `data/data.geojson`. Codelists, accuracy info, attribution,
and metadata block are regenerated each run from the live GDB.

Expected runtime: ~5–10 s on a modern machine. The first run also
fetches the CHENyx06 NTv2 grid from PROJ's CDN (`PROJ_NETWORK=ON`
is enabled at script start) — subsequent runs hit the local cache.
