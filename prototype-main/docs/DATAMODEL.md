# Green Inventory — Data Model

This document is the contract between the source FileGDB
(`GFMBundesgaerten.gdb`), the conversion script
([`scripts/gdb_to_geojson.py`](../scripts/gdb_to_geojson.py)) and the
single output the web app consumes ([`data/data.geojson`](../data/data.geojson)).
It also covers the embedded codelists, the Swiss coordinate systems
involved, the geometry-processing rules, and the data contract the
frontend depends on.

The doc is **descriptive** — generated by reading the code and the source
GDB. When the code changes, update this file.

**Audience.** A software developer with no background in green-space
management. §1 introduces the domain and the German vocabulary. §2 gives
the conceptual model in two mermaid diagrams plus a one-row-per-entity
table. §3-§6 are the source schema, codelists, conversion script, and
output schema in reference detail. §7-§8 are the coordinate-system and
geometry-processing rules that any consumer of `data.geojson` should
know about. §9-§10 are limitations and how to re-run the pipeline.

This doc is intentionally scoped to the **data model and conversion** —
not how the frontend renders it. Layer stack, legend styling, filter UI,
and identify-on-click logic live alongside the code in `js/`.

## 1. Domain primer

The app is an inventory tool for the **Bundesgärtnerei (BG)** — the
federal nursery service inside the Swiss **BBL** (Bundesamt für Bauten und
Logistik / Federal Office for Buildings and Logistics). The BG maintains
the grounds of federal real estate: courthouses, ministries, embassies,
the BG's own facilities. Each property is a **Standort** ("site") — a
parcel with some mixture of lawn, beds, hedges, trees, paths, furniture.

For every site, surveyors catalogue three kinds of thing:

1. **The site boundary** itself — a parcel outline. One per site.
2. **Maintenance areas** inside the boundary — a lawn, a flower bed, a
   hedge, a gravel path. Polygons.
3. **Point features** — every tree, planter, bench, waste bin, bird box,
   log pile. Points.

Each area and each point is tagged with a **Profil** ("care profile") —
a value from a controlled vocabulary that says what the thing is and,
implicitly, how the BG maintains it (mowing cadence, watering, pruning).
For example *Geb.Rasen kf.* is "ornamental lawn, fine-cut"; *Wildhecke*
is "wild hedge"; *Sitzbank* is "bench". The profile drives both the
colour on the map and the grouping in the legend.

### 1.1 Why split points from polygons?

The source GDB ships green-area data in two layers — one for points,
one for polygons. They share most attributes (which site, which profile,
who maintains it, irrigation, effort multiplier, etc.) but the
geometry-type split is mirrored in **two separate profile codelists**:

* **`idPP`** — 22 *point* profiles: tree types, furniture, small structures.
* **`idPPy`** — 44 *polygon* profiles: lawn types, hedges, surfaces, water.

> ⚠️ The two codelists share integer keys but mean **completely different
> things.** Profile `1` is *Laubb. nat. grossk.* ("naturally-grown large
> deciduous tree") on points but *Geb.Rasen kf.* ("ornamental lawn,
> fine-cut") on polygons. Never join `fk_profil` across the two layers
> — the conversion always carries the geometry layer along (as
> `entity_type`) so consumers can disambiguate.

| Term | Meaning |
|---|---|
| **LV03 / LV95** | Swiss national projected coordinate systems. Units are metres. `LV95` (EPSG:2056) is the modern one (introduced 2016); `LV03` (EPSG:21781) the legacy one. The source GDB is LV03; the frontend works in WGS84 with LV95 as a side channel for surveyors. |
| **FileGDB** | ESRI's proprietary geodatabase format — a directory of binary files with a `.gdb` extension. Readable via GDAL / pyogrio. |

## 2. Overview

### 2.1 Source data model (UML)

The source GDB has three layers joined by `fk_objektbezeichnung → Objekt.OBJECTID`.
Most attribute columns are integer foreign keys into one of the 17 field-domain
codelists (`idPP`, `idPPy`, `idBa`, `idPk`, `idPv`, …).

```mermaid
classDiagram
    direction LR

    class Objekt {
        +OBJECTID : int [PK]
        +objektnummer : str
        +objektbezeichnung : str
        +Adresse : str
        +Lose : str
        +erstellungsjahr : int
        +Erfassungsdatum : datetime
        +fk_pflegeklasseObjekt : int → idPk
        +fk_eigentuemer : int → idEg
        +fk_pflegeverantwortungObjekt : int → idPv
        +kontrolle : int → idJn
        +reinigung : int → idJn
        +Shape_Length : float
        +Shape_Area : float
        Geometry MultiPolygon
    }

    class Pflegeelement_point {
        +fk_objektbezeichnung : int [FK]
        +fk_profil : int → idPP
        +fk_pflegedurchfuehrung : int → idPd
        +fk_baumart : int → idBa
        +baumart : str
        +Baumnummer : int
        +maxHoehe : float
        +aufwandsfaktor : float
        +bewaesserung : int → idBw
        +lauben : int
        +ausmass : int
        +bemerkung : str
        Geometry Point
    }

    class Pflegeelement_polygon {
        +fk_objektbezeichnung : int [FK]
        +fk_profil : int → idPPy
        +fk_pflegedurchfuehrung : int → idPd
        +fk_pflegeklasse : int → idPk
        +fk_winterdienst : int → Winterdienst
        +bewaesserung : int → idBw
        +aufwandsfaktor : float
        +maxHoehe : float
        +lauben : int
        +bemerkung : str
        +geom_Length : float
        +geom_Area : float
        Geometry MultiPolygon
    }

    Objekt "1" --> "0..*" Pflegeelement_point : fk_objektbezeichnung
    Objekt "1" --> "0..*" Pflegeelement_polygon : fk_objektbezeichnung
```

> **Gotcha:** `fk_profil` on the two child layers points to **different**
> codelists (`idPP` vs `idPPy`). Code 1 is `Laubb. nat. grossk.` for points
> but `Geb.Rasen kf.` for polygons. See §3.5.

### 2.2 Pipeline

```mermaid
flowchart TD
    GDB[(GFMBundesgaerten.gdb<br/>EPSG:21781 LV03<br/>3 layers)]
    Script["scripts/gdb_to_geojson.py<br/>reproject · clean · simplify · decode codelists"]
    Out[(data/data.geojson<br/>EPSG:4326 WGS84<br/>FeatureCollection)]
    Front["Downstream consumers<br/>(out of scope here)"]

    GDB --> Script --> Out --> Front
```

The script uses **pyogrio** for layer reads, **ctypes against GDAL** for
field-domain extraction (the public pyogrio API doesn't expose
`OGR_FldDomain_GetEnumeration`), **pyproj** for reprojection (CHENyx06 NTv2
grid for LV03→LV95, 7-parameter Helmert for LV95→WGS84), and **shapely**
for geometry validity and Douglas-Peucker simplification.

### 2.3 Output entity types

The conversion fans the three source layers out into **six** `entity_type`
values. Two of the splits are non-obvious:

* `Objekt` produces **two** output features per row — a polygon (`site`)
  and a representative-point centroid (`site_location`) — so the zoomed-out
  "where are the parcels?" view has something to render.
* `Pflegeelement_point` is split on species: rows with `baumart` text or
  `fk_baumart > 0` become `tree`; everything else (furniture, structures)
  becomes `point`. Rows with a null `fk_profil` are dropped.
* `Pflegeelement_polygon` is split on shape: polygons that pass the
  circularity heuristic (§8.3) become `tree_canopy`; the rest become
  `area`. Rows with a null `fk_profil` are dropped.

```mermaid
flowchart LR
    O["Objekt"]
    PPy["Pflegeelement_polygon"]
    PPp["Pflegeelement_point"]

    O --> SITE["site<br/>(polygon)"]
    O -. centroid .-> SL["site_location<br/>(point)"]

    PPy -- non-circular --> AREA["area<br/>(polygon)"]
    PPy -. circular .-> CAN["tree_canopy<br/>(polygon)"]

    PPp -- baumart set --> TREE["tree<br/>(point)"]
    PPp -- no species --> PT["point<br/>(point)"]
```

#### Entity overview

Each output feature carries `entity_type` (machine key), `feature_type`
(German display label), `subtype` (specific label such as a species or
profile name), and the full `site_*` context block mirrored from its
parent `Objekt`.

| `entity_type` | Geometry | Source | Profile codelist | Description |
|---|---|---|---|---|
| `site` | Polygon | `Objekt` | — | Parcel / site boundaries (Standorte). Carry the full `Objekt` attribute set: address, Los (DLZ district), Eigentümer, Pflegeklasse, control/cleaning flags. |
| `site_location` | Point | `Objekt` centroid | — | Generated representative-point per site for the zoomed-out overview. Carries the same attributes as `site`. |
| `area` | Polygon | `Pflegeelement_polygon` (non-circle) | `idPPy` | Pflegeflächen — Rasen, Wiesen, Rabatten, Hecken, Gehölzflächen, Spez. Bepflanzungsformen, Beläge, Wasserflächen, Anderes. |
| `tree_canopy` | Polygon | `Pflegeelement_polygon` (circle) | `idPPy` | Polygons that pass the circularity heuristic. Mostly decorative shrub circles (Profil 19 "Gehölzrabatte", Profil 21 "Gehölz & Bodend.") — **not** literal tree crowns despite the name. Carry `crown_radius_m` / `crown_diameter_m`. |
| `tree` | Point | `Pflegeelement_point` (species set) | `idPP` | Tree positions where `baumart` or `fk_baumart` is populated. Carry species (`baumart`), `baumnummer`, `max_hoehe_m`. |
| `point` | Point | `Pflegeelement_point` (no species) | `idPP` | Non-tree point features — Abfalleimer, Sitzbank, Spielgerät, Ausstattung, Asthaufen, Baumstamm, Steinhaufen, Wildbienenhotel, Nistkasten, Pflanzgefässe, Schling-/Kletterpflanzen, Solitärstrauch, Anderes. |

Consumers typically partition these into two business-level groups:
**Standorte** = `site`, **Grünflächen** = everything else. The
`site_location` type is a rendering aid (it duplicates `site` as a
point) and is typically hidden in tabular views.

## 3. Source GDB

Path: `Daten_Bundesgärtnerei/GFMBundesgaerten.gdb` (ESRI FileGDB v10.x).

**CRS:** `EPSG:21781` (CH1903 / LV03, Bessel ellipsoid). Verified at
runtime — the script asserts the source CRS matches one of the
expected variants (LV03 or LV95) before proceeding.

### 3.1 Layers

| Layer | Geometry | Purpose |
|---|---|---|
| `Objekt` | MultiPolygon | Site / parcel boundaries (Standorte) |
| `Pflegeelement_point` | Point | Tree positions + furniture + small structures |
| `Pflegeelement_polygon` | MultiPolygon | Vegetation + surfaces + tree-canopy circles |

### 3.5 Critical insight: `fk_profil` is two distinct domains

`fk_profil` on `Pflegeelement_point` references **`idPP`**
(23 entries: trees + small structures + furniture).
`fk_profil` on `Pflegeelement_polygon` references **`idPPy`**
(44 entries: vegetation + surfaces + water).

Profile **1** is `Laubb. nat. grossk.` for points but `Geb.Rasen kf.`
for polygons. The two domains share integer codes but mean different
things — never merge them. The conversion script and frontend keep
them separate via `entity_type`.

## 4. Codelists

Extracted from the GDB's own field-domain catalog at conversion time
via `ctypes` against pyogrio's bundled GDAL DLL (the public Python
binding doesn't expose `OGR_FldDomain_GetEnumeration`). All 17 domains
are embedded in the output GeoJSON as `metadata.codelists` so the
frontend (and any downstream consumer) can introspect them without
re-querying the GDB.

### 4.1 Inventory

| Domain | Type | Entries | Fields that reference it |
|---|---|---:|---|
| `idPP` | coded | 22 | `Pflegeelement_point.fk_profil` |
| `idPPy` | coded | 44 | `Pflegeelement_polygon.fk_profil` |
| `idBa` | coded | 430 | `Pflegeelement_point.fk_baumart` |
| `idPd` | coded | 9 | `fk_pflegedurchfuehrung` (both layers) |
| `idPk` | coded | 4 | `fk_pflegeklasse*` (all layers) |
| `idPv` | coded | 4 | `fk_pflegeverantwortung*` (all layers) |
| `idJn` | coded | 3 | `kontrolle`, `reinigung` |
| `idEg` | coded | 3 | `Objekt.fk_eigentuemer` |
| `idBw` | coded | 3 | `bewaesserung` |
| `Winterdienst` | coded | 2 | `Pflegeelement_polygon.fk_winterdienst` |
| `idZs` | coded | 1 | `fk_zustand` — only `0 → "nicht erfasst"` defined. Effectively unused. |

## 5. Conversion pipeline

[`scripts/gdb_to_geojson.py`](../scripts/gdb_to_geojson.py)

### 5.1 Steps

1. **Verify source CRS.** `pyogrio.read_info(GDB, layer="Objekt")` is
   asserted to return one of `EPSG:21781` or `EPSG:2056`. Refuses to
   run otherwise — this guards against silent double-reprojection
   if a future GDB drop changes CRS.
2. **Build CRS transformers.** `_best_transformer()` enumerates
   `pyproj.transformer.TransformerGroup` and selects the lowest
   `accuracy` operation. Logs the operation chain and accuracy at
   start.
3. **Extract codelists** via `extract_codelists()` — see §4.
4. **Load `Objekt`** — each site becomes two output features (a polygon
   + a representative-point centroid). Site fields are denormalised
   onto every child feature.
5. **Load `Pflegeelement_point`** — split into trees vs other points
   based on whether `baumart` text or `fk_baumart` integer is populated.
6. **Load `Pflegeelement_polygon`** — split into tree-canopy circles vs
   regular area polygons based on a circularity heuristic (see §8.3).
7. **Assemble output** — order features site → area → canopy → tree
   → point → site_location, assign `id = array_index`.
8. **Validate + clean** — recursive walk replaces any stray NaN
   with `null`, then `json.dump(allow_nan=False)`.

### 5.2 Dependencies

```
pyogrio    0.12+   (FileGDB driver, full-schema reader)
pyproj     3.x     (CRS transforms; uses bundled CHENyx06 grid)
shapely    2.x     (geometry ops, validity, simplify, orient)
pandas     1.x     (Excel codelist read — legacy fallback only)
```

GDAL itself is bundled with pyogrio (pyogrio.libs); we load it
directly via `ctypes` for the field-domain API which pyogrio doesn't
expose.

## 7. Coordinate systems & accuracy

### 7.1 Reprojection chain

The pipeline goes **LV03 → LV95 → WGS84**. Two-step instead of direct
because the LV03↔LV95 step uses the CHENyx06 grid (sub-decimetre) and
gives us a metric LV95 representation we can use for area calculations
and metric-tolerance simplification.

| Leg | Method | Published accuracy |
|---|---|---:|
| `EPSG:21781` → `EPSG:2056` (LV03 → LV95) | CHENyx06 NTv2 grid (bundled with pyproj-data) | **0.2 m** |
| `EPSG:2056` → `EPSG:4326` (LV95 → WGS84) | 7-parameter Helmert | **1.0 m** |
| Direct LV03 → WGS84 | Composed (1.5 m) | (not used) |

Sub-decimetre WGS84 would require swisstopo's Reframe Web API
(grid-based per request). For an inventory app, ~1 m is acceptable.

### 7.2 Coordinate precision in output

All WGS84 coordinates are rounded to **7 decimal places** =
~1.1 cm at 47°N latitude / ~0.8 cm in longitude. Source data is
cm-precision, so this is a faithful round-trip.

Each feature also carries Swiss-grid coordinates as properties:

| Property | Entity types | Precision |
|---|---|---|
| `lv95_east`, `lv95_north` | `tree`, `point` | 0.01 m |
| `lv95_east_centroid`, `lv95_north_centroid` | `site`, `site_location` | 0.01 m |

LV95 is the canonical reference for Swiss surveyors and BBL staff, so
the output preserves cm-precision LV95 alongside the RFC-7946 WGS84
geometry. Consumers that need to display Swiss-grid coordinates read
the properties directly instead of round-tripping through pyproj.

## 8. Geometry processing

### 8.1 Validity and orientation

Every polygon goes through `clean_polygon()` ([gdb_to_geojson.py:128](../scripts/gdb_to_geojson.py)):

1. `make_valid()` if `geom.is_valid` is false. If the result is a
   `GeometryCollection`, the largest polygon part is kept.
2. **Conditional simplification** — Douglas-Peucker at 5 cm tolerance
   in LV95 metres, applied **only when vertex count > 200**. This
   trims the digitization-noise outliers (originally up to 41 823
   vertices on a 1 060 m² polygon) while keeping normal polygons
   pixel-perfect at any zoom. Tree-canopy candidates are skipped from
   simplification (they're regular polygons by design and we don't
   want to flatten the curvature).
3. `orient(sign=1.0)` — RFC 7946 §3.1.6 right-hand rule (outer rings
   CCW, inner rings CW). Required for Turf.js / D3-geo / leaflet-pip
   point-in-polygon tests; MapLibre is forgiving but other tools aren't.

### 8.2 Effect of conditional simplification

The conditional Douglas-Peucker pass collapses a small number of
digitization-noise outliers (polygons with thousands of vertices on a
small footprint) while leaving the bulk of the data untouched. Median
polygon vertex count is unchanged; the maximum drops by roughly an
order of magnitude. Output file size is roughly 60 % of the raw GDB
geometry payload on a typical drop. Exact numbers vary per drop; the
script logs them at conversion time.

### 8.3 Tree-canopy circle detection

Polygons in `Pflegeelement_polygon` whose perimeter and area satisfy
the regular-polygon-as-circle relation `P²/(4πA) ≈ 1` are classified
as canopies. Two thresholds gate this:

```python
CANOPY_MIN_AREA_M2 = 1.5
CANOPY_CIRCULARITY_TOL = 0.08
```

The min-area check rejects ornamental Bollensteine (round paving
stones, ~0.05 m² each) which are also 12-gon circles in the source
data. The 8 % circularity tolerance accepts 12-gons (typical),
decagons, and octagons; it rejects squares (ratio 1.273).

## 9. Known limitations

Data-model concerns; rendering concerns (legend pattern fills, missing
3D mesh, etc.) are tracked in the frontend codebase, not here.

- **WGS84 accuracy bottoms out at ~1 m** because no improved grid
  exists in PROJ's database for `CH1903+ → WGS 84`. Sub-decimetre
  needs the swisstopo Reframe Web API (out of scope).
- **`fk_zustand` and `fk_kostenstelle` codelists carry only the
  "nicht erfasst" sentinel.** `idZs` and `idKs` each define a single
  entry (`0 → "nicht erfasst"`), and every row in both child layers has
  value `0`. The fields exist for schema compatibility but carry no
  information; the conversion script decodes them to `null`.
- **`idPL` codelist is truly empty** (zero entries) — likely reserved
  for a linear-feature layer that doesn't exist in this GDB drop.
- **`fk_pflegeklasse` is barely used on individual elements.** On
  polygons only a small fraction of rows is non-zero, and every observed
  non-zero value is `2` (`PK 2`); on points the field is always `0`.
  Maintenance class is effectively a site-level attribute
  (`Objekt.fk_pflegeklasseObjekt`) inherited downward.
- **`tree_canopy` entity-type label is misleading.** Most circular
  polygons identified by the heuristic are decorative shrub plantings
  (Profil 19 "Gehölzrabatte", Profil 21 "Gehölz & Bodend."), not
  actual tree crowns. Renaming to `circular_shrub` or similar is a
  backlog item.
- **Several `fk_profil` codes have low or zero counts** (`Strap./Sportr.`,
  `Sand`, `Kunstrasen`, `Steildach`, `Dachterrasse`, some furniture
  points, `Nistkasten`) in the current GDB drop — they exist in the
  codelist but aren't well-represented in the data.

## 10. Re-running the pipeline

```sh
python scripts/gdb_to_geojson.py
```

Re-reads the GDB at `Daten_Bundesgärtnerei/GFMBundesgaerten.gdb`
(path hardcoded — change `GDB_PATH` constant if different),
overwrites `data/data.geojson`. Codelists, accuracy info, attribution,
and metadata block are regenerated each run from the live GDB.

Expected runtime: ~5–10 s on a modern machine. The first run also
fetches the CHENyx06 NTv2 grid from PROJ's CDN (`PROJ_NETWORK=ON`
is enabled at script start) — subsequent runs hit the local cache.
